  • Factory Pattern:

Single-Responsibility Principle:

Every operations class performs exactly one purpose, and calculator merely stores the results of all the called operations.


Open-Closed Principle:

The Calculator class is freely open to add additional methods, it is modular.


Liskov Substitution Principle:

Subtypes should be substitutable for base types, currently there are no sub types that run into a problem here.


Interface Segregation Principle:

There are no methods of interfaces that are not used.


Dependency Inversion Principle:

There are no entities that depend on concretions over abstraction.


Factory Pattern:

The objects are created without exposing the creation logic to the client. The client interacts with the calculator class, but the processing logic is hidden within the operation classes.
